第四个问题：

%9s 限制最多读取 9 个字符，并在字符串末尾自动添加空字符 \0，使其成为合法的 C 字符串。
fgets(arr, sizeof(arr), stdin)，如果char arr[10]的话，也是限制最多读取九个字符
普通的 scanf(“%s”, arr) 不会限制输入的长度，如果用户输入的字符串超出了 arr 的大小，会导致缓冲区溢出，从而引发未定义行为甚至安全漏洞。

实际场景

1.隐藏实现细节：通过函数访问全局变量，可以避免直接暴露 FILE 的实现细节，同时支持更多的内部优化。
2.兼容性：__acrt_iob_func 提供了一种在新版本运行时库中访问标准流的方法。
3.调试符号：如果你在 Visual Studio 调试器中查看 stdin、stdout 或 stderr，可能会看到它们通过 __acrt_iob_func 被解析。

潜在问题

1.不要直接调用 __acrt_iob_func：
它是微软的内部实现细节，不属于标准 C 函数，也没有跨平台支持。
如果需要标准流，直接使用 stdin、stdout 和 stderr 即可。
2.兼容性问题：
某些旧的二进制文件或库可能依赖旧版 CRT 的全局变量，而不是新的函数式接口。
使用 __acrt_iob_func 时，需确保链接正确版本的运行时库。

第九个问题：

调用strtok函数对字符串进行分割：

为什么 strtok(NULL, “ “)：

  • strtok 在第一次调用时，接收一个待分割的字符串，比如strtok(str, “ “)，它会返回第一个由空格分割出的子字符串，并且会将原始字符串中的空格替换成 \0，以表示子字符串的结束。
  • 为了继续分割同一个字符串，我们在后续的调用中将第一个参数设置为 NULL。这告诉strtok：从上次分割的位置继续操作，而不需要重新指定字符串。
  • NULL 告诉 strtok，它应该继续在原始字符串中查找下一个分隔符，然后返回下一个分割出的子字符串。
  • 我们第一次已经将str字符串给strtok了，故而不需要再给它了，
    用NULL即可。
